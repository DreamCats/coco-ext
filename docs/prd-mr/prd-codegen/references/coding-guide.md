# 编码执行指南

## 核心原则

**不要创新，照着样例写。**

plan.md 已经告诉你改什么、怎么改、参考什么。你的任务是执行，不是设计。

## 编码前准备

### 1. 读取上下文清单

每个文件编码前，按以下顺序读取上下文：

```
1. plan.md → 该文件的改动描述 + 上下文 + 参考样例
2. conventions.md → 命名、错误处理、日志、import 规范
3. 参考样例的实际代码 → 学习模式
4. 目标文件的当前内容 → 知道在哪里插入/修改
```

### 2. 确认改动点

在目标文件中找到要改动的位置：
- plan.md 说"在 ConvertXxx 函数中"→ 先找到这个函数
- plan.md 说"第 45 行附近"→ 定位到那个区域
- plan.md 说"参考 ConvertYyy 的写法"→ 先读 ConvertYyy

## 编码规则

### 风格一致性

- **命名**：与同文件/同模块中已有代码保持一致
- **错误处理**：照搬同文件中已有的错误处理模式
- **日志**：使用同文件中已有的日志包和格式
- **Import**：新增 import 放在正确的分组位置
- **注释**：导出函数必须加注释，风格与同文件一致

### 代码位置

- **新增字段**：放在 struct 中语义相近字段的旁边
- **新增函数**：放在同类函数的附近（按字母序或功能分组）
- **新增 import**：放在正确的分组中（标准库/公司库/项目内部）

### 不要做的事

- ❌ 不要添加 plan 中没有的改动
- ❌ 不要重构周围的代码（"顺手优化一下"）
- ❌ 不要改变现有代码的风格
- ❌ 不要添加 TODO 注释（除非 plan 要求）
- ❌ 不要注释掉旧代码（直接改）
- ❌ 不要添加 debug 日志

## 新增文件 vs 修改文件

### 新增文件

```go
// 文件头注释（如果同模块其他文件有的话）
package {包名}

import (
    // 按分组规则
)

// {导出函数注释}
func Xxx() {
    // ...
}
```

- package 声明必须与目录一致
- 文件名用 snake_case
- 导出函数/struct 必须有注释

### 修改文件

- 先读完整个文件（或至少读到改动点的上下文）
- 找到插入/修改位置
- 最小化改动范围 — 只改 plan 要求的部分
- 保持缩进和空行风格与原文件一致

## 增量编译

**每个文件改完立即编译**，不要等到全部改完。

```bash
# 编译改动的 package
go build ./{package}/...

# 如果编译通过，继续下一个文件
# 如果编译失败，立即修复（参见 build-fix.md）
```

为什么要增量编译：
- 早发现问题，修复成本低
- 避免多个文件的错误叠加，难以定位
- 编译通过的文件可以作为后续文件的依赖

## 代码量控制

- 单个文件的新增代码通常 ≤ 50 行
- 如果一个文件需要加 100+ 行，检查是否应该拆分
- 超出 plan 预期的改动量 → 暂停并告知用户
