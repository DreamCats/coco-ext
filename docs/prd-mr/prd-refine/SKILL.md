# /prd-refine - PRD 理解与完善

**在代码调研之前，先把 PRD 本身搞清楚。**

很多 PRD 写得模糊、缺边界条件。带着模糊的 PRD 去做代码调研，评估不准，后期返工成本高。本 skill 的目标：主动发现 PRD 盲点，通过探索式问答补齐，输出一份高质量的 refined PRD。

## 触发方式

```bash
# 直接传入需求描述
/livecoding:prd-refine --prd "在讲解卡上增加倒计时展示"

# 传入飞书 PRD 链接
/livecoding:prd-refine --prd https://bytedance.larkoffice.com/docx/xxx

# 传入本地文件
/livecoding:prd-refine --prd ./prd-draft.md
```

## 产物

写入 `.livecoding/tasks/{task-id}/` 目录：

| 文件 | 内容 |
|------|------|
| `prd.md` | PRD 原文（存档） |
| `prd-refined.md` | 完善后的 PRD（含打分 + 问答记录） |

**task-id 格式**：`YYYYMMDD-简述`（如 `20260310-add-countdown`），从 PRD 内容自动生成。

## 核心流程

```
读取 PRD 输入（文本 / 飞书链接 / 本地文件）
      │
      ├─ Step 0: PRD 输入解析
      │    └─ 飞书链接 → 转 Markdown
      │    └─ 本地文件 → 读取内容
      │    └─ 纯文本 → 直接使用
      │
      ├─ Step 1: 结构化解析
      │    └─ 提取功能点、涉及的业务概念、预期行为
      │
      ├─ Step 2: 5 维质量打分（PM 视角）
      │    └─ 参考打分模型，给出每个维度的分数 + 理由
      │
      ├─ Step 3: 探索式问答（如果总分 < 20）
      │    └─ 针对低分维度生成问题
      │    └─ RD 回答 → 合并到 PRD → 重新打分
      │    └─ 循环直到达标或满 3 轮或 RD 说"够了"
      │
      └─ Step 4: 输出 prd-refined.md
           └─ 完善后的 PRD + 打分 + 问答记录
```

## Step 0: PRD 输入解析

### 飞书链接

检测 `$ARGUMENTS` 是否包含飞书/Lark 链接（`feishu.cn` 或 `larksuite.com`）：

```bash
node "$CLAUDE_PLUGIN_ROOT/scripts/livecoding/lark-import.js" \
  --url "$ARGUMENTS" \
  --project-root "$(git rev-parse --show-toplevel)"
```

导入成功后基于 Markdown 内容继续。

### 本地文件

如果参数是文件路径（`.md`、`.txt`），直接读取内容。

### 纯文本

直接使用用户输入的文本作为 PRD。

### 存档

将 PRD 原文写入 `.livecoding/tasks/{task-id}/prd.md`。

## Step 1: 结构化解析

从 PRD 原文中提取关键信息：

```markdown
### PRD 结构化摘要

- **需求概述：** 一句话概括要做什么
- **功能点：**
  - 功能 1：xxx
  - 功能 2：xxx
- **涉及的业务概念：** 讲解卡、倒计时、拍卖
- **涉及的页面/入口：** 讲解卡底部区域
- **预期行为：** 新增 UI 展示 / 新增接口 / 修改现有逻辑
```

**如果 glossary 可用**（`.livecoding/context/glossary.md` 存在），同时做术语匹配：
- 将 PRD 中的业务术语映射为代码标识符
- 标记 glossary 中找不到的新术语（提醒后续补充）

## Step 2: 5 维质量打分

**参考**：[打分模型](references/scoring-model.md)

**核心原则：只评价 PM 应该写清楚的内容，不评价技术实现细节。**

| 维度 | 评估内容 | 权重 |
|------|---------|------|
| **功能明确度** | 要做什么功能，描述是否具体 | 必评 |
| **边界完整度** | 异常/空状态/极端情况是否覆盖 | 必评 |
| **交互/展示** | UI 表现、操作行为是否说清 | 必评 |
| **验收标准** | 怎么算做完，QA 怎么验 | 必评 |
| **业务规则** | 业务逻辑与约束条件 | 必评 |

每个维度 1-5 分，总分 5-25 分。

### 打分输出格式

```markdown
📋 PRD 质量评估

PRD：{需求概述}
总分：{N}/25

| 维度 | 分数 | 评价 |
|------|------|------|
| 功能明确度 | {1-5}/5 | {一句话理由} |
| 边界完整度 | {1-5}/5 | {一句话理由} |
| 交互/展示 | {1-5}/5 | {一句话理由} |
| 验收标准 | {1-5}/5 | {一句话理由} |
| 业务规则 | {1-5}/5 | {一句话理由} |
```

### 总分决策

| 总分 | 等级 | 决策 |
|------|------|------|
| **20-25** | ✅ 清晰 | 直接进入 Step 4 输出 |
| **12-19** | ⚠️ 有缺失 | 进入 Step 3 探索式问答 |
| **5-11** | ❌ 太粗 | 提示需大幅补充，列出所有缺失项，仍进入 Step 3 |

## Step 3: 探索式问答

**参考**：[问答策略](references/qa-strategy.md)

### 问题生成原则

- ✅ **只问 PM/RD 能回答的**：业务规则、交互行为、边界条件、验收标准
- ❌ **不问技术实现**：不问"数据从哪个字段来"、"调哪个 RPC"、"用什么数据结构"
- 🎯 **只针对低分维度**：≥ 4 分的维度不提问
- 📝 **问题要具体**：不说"请补充边界条件"，而说"倒计时结束后怎么处理？隐藏还是显示已结束？"

### 问答输出格式

```markdown
📋 PRD 质量评估

PRD：在讲解卡上增加倒计时展示
总分：14/25

✅ 功能明确度：4/5 — 清楚，在讲解卡上加拍卖倒计时
⚠️ 边界完整度：2/5 — 以下问题未说明：
   Q1: 倒计时结束后怎么处理？隐藏？显示"已结束"？
   Q2: 没有拍卖活动的讲解卡，倒计时区域怎么处理？
⚠️ 交互/展示：2/5 —
   Q3: 倒计时展示在讲解卡的哪个位置？有设计稿吗？
   Q4: 需要动画效果吗？（比如最后 10 秒闪烁）
✅ 验收标准：4/5 — 基本清楚
⚠️ 业务规则：2/5 —
   Q5: 所有讲解卡都展示，还是仅关联拍卖活动的？
   Q6: 买家端和主播端都展示吗？

请回答以上问题，我会更新 PRD 后重新评估。
```

### 问答循环

1. **RD 回答后**：将答案合并进 PRD 结构化摘要
2. **重新打分**：基于补充后的内容重新 5 维评估
3. **判断是否继续**：
   - 总分 ≥ 20 → 结束问答，进入 Step 4
   - 总分 < 20 且轮次 < 3 → 继续提问（只问仍低分的维度）
   - 总分 < 20 且轮次 = 3 → 告知剩余缺失，结束问答
   - RD 说"够了"/"继续"/"跳过" → 立即结束问答
4. **每轮问题数量**：3-6 个，不要一次丢 10+ 个问题

### 合并策略

RD 的回答按维度归类，合并到 PRD 对应部分：

```
Q: 倒计时结束后怎么处理？
A: 显示"已结束"，3s 后隐藏

→ 合并到「边界条件」部分：
  "倒计时结束后显示'已结束'，3 秒后自动隐藏倒计时区域"
```

## Step 4: 输出 prd-refined.md

**参考**：[产物模板](references/refined-template.md)

将最终结果写入 `.livecoding/tasks/{task-id}/prd-refined.md`。

完成后输出摘要：

```markdown
## ✅ PRD 完善完成

- **任务目录：** .livecoding/tasks/{task-id}/
- **初始分：** {N}/25
- **完善后：** {M}/25
- **问答轮次：** {K} 轮
- **遗留问题：** {有/无}

### 下一步
- 确认 prd-refined.md 内容无误
- 执行 `/livecoding:prd-assess` 进入代码调研与评估
```

## 注意事项

1. **不要替 PM 做决定**：边界条件、业务规则的答案必须来自人，AI 只负责提问
2. **不问技术问题**："数据从哪来"、"调哪个接口"是 Phase A 的事
3. **问题要可回答**：避免开放式大问题（"你觉得还有什么要补充的？"），问具体的 yes/no 或选择题
4. **尊重 RD 的时间**：如果 RD 说"够了"、"差不多了"、"先这样"，立即结束问答
5. **术语记录**：如果 PRD 中出现 glossary 里没有的新术语，在 prd-refined.md 末尾标记，提醒补充
6. **不评价 PRD 质量**：打分是为了发现缺失，不是批判 PM 写得差。语气客观中立
7. **飞书链接容错**：如果飞书导入失败，提示用户直接粘贴 PRD 文本

## 与其他 skill 的关系

```
context init / update → 生成 context/glossary.md
                    ↓ 辅助术语匹配
              prd-refine（本 skill）
                    ↓ 输出 prd-refined.md
              prd-assess → 读 prd-refined.md 做代码调研
                    ↓
              prd-codegen → 按 plan.md 编码
```

**prd-refine 确保 PRD 质量达标后再进入技术评估，避免带着模糊需求做代码调研。**
