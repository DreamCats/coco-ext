# PRD 质量打分模型

## 原则

**只评价 PM 应该写清楚的内容，不评价技术实现细节。**

"数据从哪个字段来"、"调哪个 RPC"是 RD 的事，不在评分范围内。

## 5 维评分标准

### 1. 功能明确度

> 要做什么功能，描述是否具体可执行

| 分数 | 标准 | 示例 |
|------|------|------|
| 1 | 只有一个模糊方向 | "优化用户体验" |
| 2 | 知道要做什么，但太笼统 | "加个倒计时" |
| 3 | 有具体功能描述，但缺细节 | "在讲解卡上加倒计时" |
| 4 | 功能描述清晰，基本可执行 | "在讲解卡底部增加拍卖倒计时，实时递减" |
| 5 | 功能描述完整，无歧义 | "在讲解卡底部增加拍卖倒计时，格式 HH:MM:SS，每秒刷新" |

### 2. 边界完整度

> 异常情况、空状态、极端条件是否考虑

| 分数 | 标准 | 示例 |
|------|------|------|
| 1 | 完全没提边界条件 | — |
| 2 | 提了 1-2 个，但明显遗漏 | "倒计时结束后隐藏"（没说无拍卖时怎么处理） |
| 3 | 主要路径覆盖，部分边界缺失 | 覆盖了正常/结束，漏了并发/极端 |
| 4 | 大部分边界都考虑了 | 正常/结束/无数据/异常 都提了 |
| 5 | 边界条件完整，包括极端情况 | 还考虑了时区、跨天、网络异常等 |

**常见需要考虑的边界：**
- 数据为空时（无拍卖 / 无商品 / 无数据）
- 状态结束时（倒计时结束 / 活动过期）
- 异常情况（接口超时 / 数据不一致）
- 并发/竞态（同时多个操作）
- 极端值（超长文本 / 超大数量）

### 3. 交互/展示

> UI 表现、操作行为是否描述清楚

| 分数 | 标准 | 示例 |
|------|------|------|
| 1 | 完全没提 UI/交互 | — |
| 2 | 只提了"加个xxx" | "加个标签" |
| 3 | 提了位置或样式之一 | "在底部加标签" |
| 4 | 位置+基本样式都提了 | "底部，红色标签" |
| 5 | 位置+样式+交互行为完整 | "底部价格右侧，红色 #FF4444，14px，点击跳转详情" |

**注意：**
- 如果有设计稿/UI 图，交互/展示维度可直接给 4-5 分
- 纯后端接口需求（无 UI），此维度给 N/A，不计入总分
- N/A 时总分按 4 维计算：满分 20，阈值按比例调整（达标 = 16）

### 4. 验收标准

> 怎么算做完了，QA 怎么验证

| 分数 | 标准 | 示例 |
|------|------|------|
| 1 | 没提验收标准 | — |
| 2 | 模糊的"能用就行" | "倒计时能展示" |
| 3 | 有基本验收条件 | "倒计时能正常展示和结束" |
| 4 | 验收条件较完整 | "倒计时精确到秒，结束后刷新状态" |
| 5 | 可直接写测试用例 | "拍卖中→显示倒计时；结束→显示已结束 3s 后隐藏；无拍卖→不显示" |

### 5. 业务规则

> 业务逻辑、约束条件、生效范围

| 分数 | 标准 | 示例 |
|------|------|------|
| 1 | 没有业务规则描述 | — |
| 2 | 只有隐含的规则 | "在讲解卡上"（隐含只有讲解卡） |
| 3 | 部分业务规则明确 | "仅关联拍卖的讲解卡展示" |
| 4 | 主要规则清楚 | "关联拍卖 + 仅买家端" |
| 5 | 规则完整且无歧义 | "仅已上架且关联拍卖的商品，仅买家端可见，主播端不展示" |

**常见业务规则维度：**
- 生效范围（哪些用户/角色/场景可见）
- 前置条件（什么条件下触发）
- 互斥规则（与其他功能的冲突）
- 优先级（多个规则冲突时哪个优先）
- 时效性（是否有时间限制）

## 纯后端需求的调整

如果 PRD 是纯后端/接口需求（无 UI），调整如下：

- **交互/展示** → 改为 **接口规格**：入参、出参、错误码是否说清
- 评分标准不变，只是关注点从 UI 变为接口契约
- 或者标记 N/A，按 4 维打分

## 打分注意事项

1. **相对打分**：同一维度内，分数之间要有区分度
2. **给分理由**：每个维度必须写一句话理由
3. **不评判 PM**：打分是为了发现缺失，不是批评 PRD 质量
4. **宽容向上**：如果某个信息能从上下文合理推断，给 +0.5 的倾向
