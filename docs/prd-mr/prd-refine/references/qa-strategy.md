# 探索式问答策略

## 核心原则

问答的目标是**补齐 PRD 缺失的信息**，不是考试。

### 问什么

| ✅ 应该问 | ❌ 不应该问 |
|-----------|-------------|
| 业务规则：谁能看到？什么条件下展示？ | 技术选型：用 Redis 还是 DB？ |
| 边界条件：数据为空时怎么处理？ | 实现方式：调哪个 RPC？ |
| 交互行为：点击后跳转哪里？ | 数据来源：从哪个字段读？ |
| 验收标准：怎么算做完？ | 性能指标：QPS 要求多少？ |
| UI 展示：颜色、位置、动画？ | 代码结构：放在哪个 package？ |

### 怎么问

**好问题的特征：**
- 🎯 **具体**：不说"请补充边界条件"，而说"倒计时结束后展示什么？"
- 🔲 **可回答**：yes/no、选择题、或一句话能说清的
- 📐 **有选项**：给出 AI 的猜测选项，让人确认或纠正

**示例对比：**

| 差的问法 | 好的问法 |
|---------|---------|
| "请补充边界条件" | "倒计时结束后怎么处理？隐藏倒计时区域？还是显示'已结束'？" |
| "交互是什么样的" | "倒计时展示在讲解卡的哪个位置？底部？标题旁边？有设计稿吗？" |
| "还有什么要补充的吗" | "买家端和主播端都展示倒计时吗？还是只有买家端？" |

### 问题分组

按维度分组展示，每组 1-3 个问题：

```markdown
⚠️ 边界完整度（2/5）：
   Q1: 倒计时结束后怎么处理？隐藏？还是显示"已结束"？
   Q2: 没有关联拍卖活动的讲解卡，倒计时区域是否展示？

⚠️ 交互/展示（2/5）：
   Q3: 倒计时展示在讲解卡的哪个位置？有设计稿吗？
```

### 问题数量控制

| 总分 | 低分维度数 | 建议问题数 |
|------|----------|-----------|
| 15-19 | 1-2 个 | 3-4 个问题 |
| 10-14 | 2-3 个 | 4-6 个问题 |
| 5-9 | 3-5 个 | 5-6 个（上限，不要超过 6 个） |

**超过 6 个问题时：** 只问最关键的 6 个，其余标记为"如果方便也请补充"。

## 答案合并

### 合并规则

1. 将答案按维度归类到 PRD 对应部分
2. 用 RD 的**原始表述**（不要过度改写）
3. 如果答案补充了新的业务概念，记录到"新术语"部分

### 合并示例

```
Q: 倒计时结束后怎么处理？
A: 显示"已结束"，3 秒后自动隐藏

→ 合并到 PRD「边界条件」：
  - 倒计时结束后显示"已结束"，3 秒后自动隐藏倒计时区域

Q: 买家端和主播端都展示吗？
A: 都展示

→ 合并到 PRD「业务规则」：
  - 买家端和主播端都展示倒计时
```

### 冲突处理

如果 RD 的回答与 PRD 原文矛盾：
- 以 RD 回答为准（PRD 可能没更新）
- 在问答记录中标注"与原 PRD 不一致"

## 循环终止条件

| 条件 | 动作 |
|------|------|
| 总分 ≥ 20 | 结束问答，输出 refined PRD |
| 轮次 = 3 | 告知剩余缺失项，结束问答 |
| RD 说"够了/继续/跳过/差不多了" | 立即结束问答 |
| 新一轮与上一轮分数相同 | 结束问答（再问也补不了更多了） |

**第 3 轮结束时的输出：**

```markdown
已完成 3 轮问答，以下信息仍未确认，建议后续补充：
- [ ] xxx
- [ ] yyy

当前分数足以进入代码调研，遗留问题会记录在 prd-refined.md 中。
```

## 特殊情况

### PRD 特别短（< 20 字）

如"加个倒计时"，很可能是 RD 口头描述而非正式 PRD：
- 正常打分（大概率低分）
- 问答时语气更友好，把它当作"需求讨论"而非"PRD 评审"

### PRD 特别长（> 2000 字）

- 先做摘要，提取关键功能点
- 打分时按功能点拆分评估
- 如果有多个功能点，建议拆分成多个 task

### 纯后端需求

- "交互/展示"维度改为评估"接口规格"
- 或标记 N/A，总分按 4 维计算（满分 20，达标 16）
