# 复杂度评估模型

## 5 维技术视角

### 1. 改动范围

| 🟢 简单 (1) | 🟡 中等 (2) | 🔴 复杂 (3) |
|-------------|-------------|-------------|
| 1-3 个文件 | 4-10 个文件 | 10+ 个文件 |
| 单 package | 跨 2-3 个 package | 跨服务 / 跨仓库 |
| 改动集中 | 改动分散但有规律 | 改动分散且关联复杂 |

**判断方法：** 数代码调研中列出的文件数，看跨几个 package。

### 2. 接口变更

| 🟢 简单 (1) | 🟡 中等 (2) | 🔴 复杂 (3) |
|-------------|-------------|-------------|
| 不改 IDL | 新增接口，模式类似已有的 | 修改已有接口签名 |
| 只加 response 字段 | 加 request + response 字段 | 改 request 已有字段语义 |
| 不影响调用方 | 新增接口需通知调用方 | 已有接口变更需所有调用方适配 |

**风险标记：** 改已有接口签名是高风险操作，即使评估为中等也要特别标注。

### 3. 数据模型

| 🟢 简单 (1) | 🟡 中等 (2) | 🔴 复杂 (3) |
|-------------|-------------|-------------|
| 不动表 | 新建表，关系简单 | 改已有表结构 |
| 只读已有字段 | 加列（nullable，有默认值） | 需要数据迁移 |
| 不涉及存储 | 加索引 | 改已有列类型/约束 |

**注意：** "加列"如果是 nullable + 有默认值 → 🟢；如果是 NOT NULL 且无默认值 → 🟡（需要处理存量数据）。

### 4. 业务逻辑

| 🟢 简单 (1) | 🟡 中等 (2) | 🔴 复杂 (3) |
|-------------|-------------|-------------|
| CRUD / 透传 / 格式转换 | 条件分支（2-5 个分支） | 多条件组合（5+ 分支） |
| 单一数据流 | 状态流转（2-3 个状态） | 状态机（4+ 状态 + 转移条件） |
| 无并发考虑 | 简单的读写锁 | 并发事务 / 分布式锁 |

**判断方法：** 看 PRD 中的业务规则数量和条件复杂度。

### 5. 依赖关系

| 🟢 简单 (1) | 🟡 中等 (2) | 🔴 复杂 (3) |
|-------------|-------------|-------------|
| 不依赖新服务 | 依赖 1-2 个已有下游 | 依赖新服务（需联调） |
| 数据已有 | 需要调已有 RPC 新接口 | 需要新建 RPC 接口 |
| 无跨团队协作 | 需了解下游返回格式 | 需跨团队联调 |

## 总分计算

直接求和：总分 = 维度1 + 维度2 + 维度3 + 维度4 + 维度5

| 总分 | 等级 | 决策 |
|------|------|------|
| **5-7** | ✅ 简单 | 生成完整编码计划 |
| **8-10** | ⚠️ 中等 | 生成编码计划，标记风险点 |
| **11-15** | ❌ 复杂 | 只输出调研结果，不生成编码计划 |

## 评估注意事项

1. **宁可高估不要低估** — 不确定时给高分，避免 Phase B 执行到一半发现太复杂
2. **关注连锁反应** — 改一个 struct 可能影响所有引用方
3. **IDL 变更是放大器** — 改 IDL 意味着需要重新生成代码 + 通知所有调用方
4. **区分"读"和"写"** — 读已有数据 = 简单；写/改已有数据 = 复杂度上升
5. **跨服务 = 自动加分** — 涉及多个服务时，每个额外服务至少 +1

## 边界情况

### 总分刚好 7（简单上限）
- 仔细 review 每个维度是否给低了
- 如果有任何一个维度不确定，升为中等

### 总分刚好 10（中等上限）
- 如果有 2 个以上维度是 🔴，建议直接判为复杂
- 单个 🔴 + 其他 🟡 → 保持中等，但标记关键风险

### 纯新增（不改已有代码）
- 新增文件/接口/表 → 通常比改已有的简单
- 但新增量大时（10+ 新文件）仍然是中等或复杂
