## 填写规则

### 改动顺序

推荐按依赖关系排列（被依赖的先改）：

1. **IDL / protobuf**（如果需要改）
2. **model / struct 定义**
3. **converter / 数据转换**
4. **service / 业务逻辑**
5. **handler / 入口**
6. **测试文件**

### 改动描述的精度

**不够具体（差）：**
> 在 converter 中加字段映射

**足够具体（好）：**
> 在 ConvertPopCardResponse() 函数中（第 45 行附近），参考 ConvertPopCardPrice() 的写法，
> 新增 Countdown 字段映射：resp.Countdown = rpcResp.AuctionConfig.GetCountdown()

### 工作分支命名

格式：`{type}/{简述}-{YYYYMMDD}`

| type | 场景 |
|------|------|
| feat | 新功能 |
| fix | 修 bug |
| refactor | 重构 |
| perf | 性能优化 |

示例：`feat/add-auction-countdown-20260310`

### 状态流转

```
draft → approved → completed
  ↑        ↓          ↓
  └── rejected    (Phase B 完成后自动标记)
```

- `draft`：AI 生成，待人确认
- `approved`：人确认后修改此字段（Phase B 检查此状态）
- `rejected`：人认为计划不对，需要修改后重新评估
- `completed`：Phase B 执行完毕后自动标记

### 简单 vs 中等的区别

| | 简单（5-7） | 中等（8-10） |
|---|---|---|
| 改动描述 | 精确到函数和行号 | 精确到函数，行号可能不准 |
| 参考样例 | 有高度相似的样例 | 样例相似度中等，需适配 |
| 风险点 | 无或极少 | 每个不确定的地方都标记 |
| ⚠️ 人工关注 | 无此 section | 必须列出需要人确认的点 |
| 编码信心 | AI 大概率一次写对 | AI 可能需要人指导部分写法 |
