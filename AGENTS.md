## 编码规范

- 必须保持现有行为和配置不变，除非用户明确要求修改
- 应优先使用清晰、明确的 `if/else`，避免嵌套三元表达式
- 不应为了简短而编写影响可读性的单行代码
- 函数应保持短小、聚焦，职责应尽量单一
- 不得重构架构层级代码，除非用户明确提出该需求
- **禁止运行全局 build 命令**，例如 `go build ./...`
- **禁止运行全局 test 命令**，例如 `go test ./...`
- **必须执行最小范围验证**，只构建或验证本次改动涉及的包、模块或服务
- **禁止保留魔法值**，应将魔法数字和魔法字符串提取为具名常量
- **必须检查 nil**，在解引用指针、访问 map 值或使用接口值前先进行 nil 判断

## 注释规范

- 导出函数必须提供文档注释（Go：`// FuncName ...`）
- 复杂逻辑必须补充解释意图的行内注释
- 注释应优先说明“为什么”，避免重复代码本身已经表达清楚的“做了什么”
- 注释风格、语气和格式应与仓库现有代码保持一致
